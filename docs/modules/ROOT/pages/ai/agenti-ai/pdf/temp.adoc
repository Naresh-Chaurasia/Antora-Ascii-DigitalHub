==== Solution

This document defines a complete MCP resource schema for a sales intelligence AI assistant.

The system enables AI-driven access to customer, deal, and activity data from a CRM platform.

##############################################

---

==== 1. Resource Inventory

The following core resources are exposed:

[cols="1,3"]
|===
| Resource | Description

| customers
| Customer profiles including company details and account status

| deals
| Sales opportunities with value, stage, and timelines

| activities
| Sales interactions such as calls, meetings, emails

| pipeline
| Aggregated pipeline data grouped by stages

| insights
| AI-generated insights based on combined data
|===

sep

==== 2. Resource URI Design

URIs follow a consistent and scalable pattern:

----
<resource>://<filter>?<query_params>
----

*Examples*

[cols="1,3"]
|===
| URI | Description

| customers://all
| Fetch all customers

| customers://high-value
| Customers with high revenue potential

| customers://{customerId}
| Specific customer details

| deals://active
| All active deals

| deals://closing-soon
| Deals closing within next 30 days

| deals://{dealId}
| Specific deal details

| activities://recent
| Recent sales activities

| activities://customer/{customerId}
| Activities for a specific customer

| pipeline://overview
| Pipeline summary grouped by stages

| insights://customer/{customerId}
| AI insights for a customer

| insights://pipeline
| AI insights on overall pipeline
|===

==== 3. Resource Schema Definitions

*Customer Resource*

[source,json]
----
{
  "id": "string",
  "name": "string",
  "industry": "string",
  "revenue": "number",
  "status": "active | inactive",
  "created_at": "datetime"
}
----

---

*Deal Resource*

[source,json]
----
{
  "id": "string",
  "customer_id": "string",
  "value": "number",
  "stage": "prospecting | negotiation | closed",
  "close_date": "date",
  "status": "open | won | lost"
}
----

---

*Activity Resource*

[source,json]
----
{
  "id": "string",
  "customer_id": "string",
  "type": "call | email | meeting",
  "timestamp": "datetime",
  "notes": "string"
}
----

---

*Pipeline Resource*

[source,json]
----
{
  "stage": "string",
  "total_deals": "number",
  "total_value": "number"
}
----

---

*Insights Resource*

[source,json]
----
{
  "type": "customer | pipeline",
  "summary": "string",
  "recommendations": ["string"]
}
----

==== 4. Access Patterns

The AI assistant uses the following patterns:

*Common Queries*

* "Show high-value customers"
  → customers://high-value

* "Deals closing this month"
  → deals://closing-soon

* "Recent activities for a customer"
  → activities://customer/{customerId}

* "Pipeline summary"
  → pipeline://overview

* "Insights for customer"
  → insights://customer/{customerId}

---

*Resource Relationships*

* customers ↔ deals (1:N)
* customers ↔ activities (1:N)
* deals → pipeline (aggregation)
* all resources → insights

---

*Combined Queries (AI Behavior)*

AI may:

* Fetch customers + deals → identify high-value opportunities  
* Fetch deals + activities → detect stalled deals  
* Fetch pipeline + insights → recommend actions

==== 5. Filtering and Query Parameters

URIs support additional filters:

----
deals://active?min_value=50000&stage=negotiation
customers://all?industry=finance
activities://recent?days=7
----

---

==== 6. Security Considerations

Each resource enforces:

* Input validation  
* Role-based access control  
* Data filtering  

Example:

* Sales Rep → only their customers  
* Manager → full access  

Sensitive data is excluded by default.

---

==== 7. Design Principles

* Keep resources small and focused  
* Use predictable URI patterns  
* Avoid overloading a single resource  
* Enable composability for AI  

---

==== 8. Example End-to-End Flow

User Query:

----
"Show high-value deals closing soon"
----

AI Execution:

1. Call deals://closing-soon  
2. Filter by value > threshold  
3. Fetch related customers  
4. Generate insights
