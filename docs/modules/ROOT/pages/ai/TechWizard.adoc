== 📍 VS Code Project: DevOps Context Assistant 

* *Python Version*

=== Overview

This document describes an intermediate, ~8-hour project to build a VS Code extension that provides DevOps/IT infrastructure help using a fully local Retrieval-Augmented Generation (RAG) setup

The LLM is GitHub Copilot (via VS Code’s `vscode.lm`), while retrieval is handled locally with a Python service and ChromaDB.

=== Target Users / Use Cases

The extension provides context-aware assistance for:

* Azure resources (App Service, Functions, AKS, Cosmos DB, Key Vault)
* Security patterns (Managed Identity, Key Vault integration, network security)
* CI/CD pipelines (Azure DevOps, GitHub Actions, deployment strategies)
* Infrastructure-as-Code (Bicep, ARM, Terraform)
* Monitoring & logging (Application Insights, Log Analytics)
* Troubleshooting (runbooks, common issues, best practices)

=== ✔️ What You Build

* Sample infrastructure documentation corpus (templates, best practices, policies)
* Local Python RAG service (FastAPI) for indexing and vector search (no LLM calls)
* VS Code extension that retrieves context from the RAG service and queries Copilot
* Fully local operation with no external API keys (Copilot subscription required)

=== Tech Stack

* VS Code extension: TypeScript
* RAG service: Python + FastAPI + LangChain (retrieval only)
* Vector DB: ChromaDB
* Embeddings: local Python embedding service using Sentence Transformers
* Infra parsing (optional/mentioned): `pybicep`, `python-terraform-parser`

=== Architecture Summary

Flow:

* User query in VS Code extension
* Extension calls local RAG service over HTTP (localhost:8000)
* RAG service generates embeddings by calling a local embedding service (localhost:5001)
* RAG service searches ChromaDB and returns top context chunks
* Extension sends context + question to GitHub Copilot via `vscode.lm` and streams response

Key points:

* No Azure/OpenAI keys required; everything is local except Copilot.
* Pattern: Retrieve context -> pass to LLM -> generate answer.

=== Sprint 1: Extension Shell (2–3 hours)

==== Step 1.1: Configure GitHub Copilot (15 min)

Purpose: create project-specific Copilot guidance so code generation follows your architecture and conventions.

Actions:

* Create workspace folder: `Devops-Context-Assistant`
* Add `.github/copilot-instructions.md` documenting:
  - Tech stack (TypeScript extension, Python FastAPI RAG, ChromaDB, `vscode.lm`)
  - Python standards (PEP8, type hints, async FastAPI, Pydantic, structlog, pathlib)
  - “Critical” LLM call pattern: call RAG first, then call Copilot via `vscode.lm`

==== Step 1.1 (continued): VS Code Workspace Settings

Purpose: ensure consistent formatting and Copilot behavior.

Actions:
* Create `.vscode/settings.json` enabling:
  - format on save
  - Prettier for TypeScript (tab size 2)
  - Black for Python (tab size 4)
  - Bicep formatter and Terraform formatter (tab size 2)
  - `github.copilot.chat.codeGeneration.useInstructionFiles: true`

==== Step 1.2: Setup Local Python Embedding Service (25–45 min)

Goal: run embeddings locally with Sentence Transformers (no keys).

Service requirements:
* FastAPI on port 5001
* Endpoints:
  - `GET /` health check
  - `POST /embeddings` with `{ "text": "..." }` -> `{ "embedding": [...], "model": "...", "dimension": 384 }`
  - `POST /embeddings/batch` for batch requests
* Model: `all-MiniLM-L6-v2` (384-dim); mention alternative `all-mpnet-base-v2` in README
* Clear logging that it’s local/no keys

Setup/run instructions (high level):
* Create venv, install requirements, run `python app.py`
* Validate with curl health check and a sample embedding request
* Keep this service running for the RAG service to use

==== Step 1.3: Create VS Code Extension (45 min)

Goal: scaffold the full extension (TypeScript) with bundling and a chat UI.

Key requirements:
* Webpack bundling -> `dist/extension.js`
* Source layout includes:
  - `extension.ts`
  - commands
  - `providers/ChatViewProvider.ts`
  - `services/RagServiceClient.ts`
  - `utils/FileTypeDetector.ts`
* Features:
  - Activity bar sidebar with webview chat
  - Commands (Ask Question, Review File, Find Pattern, Check Security)
  - Context menu actions for `.bicep`, `.tf`, `.yaml`
  - HTTP client calling the Python RAG API at `http://localhost:8000` (critical: not 5000)

=== MCP Connections

=== What is Model Context Protocol (MCP)?

*MCP* (Model Context Protocol) is a standard protocol for connecting AI assistants to external data sources.

Key points:

* Gives GitHub Copilot access to repositories, databases, and APIs
* Built-in MCP servers:
** GitHub
** GitLab
** Azure DevOps
* Supports custom MCP servers:
** Database connections
** Internal documentation
* Requires VS Code 1.99+ for full support

In practical terms, MCP allows GitHub Copilot to read from real systems instead of relying solely on its training data** This enables more accurate, relevant, and maintainable suggestions.

=== Benefits of Model Context Protocol (MCP)

* Reference real implementations instead of copy-paste code
* Always up-to-date by reading from live repositories
* Context-aware understanding of actual file structures
* Learn from working code and proven patterns
* Team knowledge sharing through internal repositories

The biggest advantage of MCP is that Copilot reasons over *real, working implementations*, not hypothetical examples.

=== Setting Up GitHub Model Context Protocol (MCP)

Steps:

** Open Command Palette (`Cmd/Ctrl + Shift + P`)
** Search for *GitHub Copilot: Manage Model Context Protocols*
** Click *Add MCP Server*
** Select *GitHub*
** Authenticate with your GitHub account (if prompted)

Verification:

* Open Copilot Chat
* Type `@github`
* Repository suggestions should appear
* A green checkmark indicates a successful connection

This is a one-time configuration per environment.

=== @Participants

=== What is a Participant?

Participants are special keywords that give Copilot scoped access to specific contexts.

Common Participants:

* `@workspace` – Access local project files
* `@github` – Access GitHub repositories via MCP
* `@terminal` – Generate or reason about terminal commands

Using Participants reduces ambiguity and significantly improves response quality.

=== Participant Usage

Standard usage pattern for GitHub:

```

@github owner/repo-name <your question or task>

```

Specifying the repository ensures Copilot reasons over the correct codebase** This is especially effective for:

* Architecture discussions
* Integration approaches
* Specific implementation details
