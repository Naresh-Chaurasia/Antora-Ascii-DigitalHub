* Model Context Protocol (MCP)
  ** MCP is a communication layer that provides an AI model with context and tools.
  ** It reduces the need for developers to write large amounts of integration code.
  ** It simplifies connecting AI models with external services.

* Core Components of MCP
  ** Client
  *** The application that interacts with the AI model.
  *** Example: A chat interface used by a user.
  ** Server
  *** Hosts integrations with external systems.
  *** Contains internal components such as tools, resources, and prompts.

* Example Use Case: GitHub Chat Application
  ** A chat interface allows a user to ask questions about their GitHub account.
  ** Example query: “What open pull requests do I have across all repositories?”
  ** The AI needs to access GitHub data to answer the question.

* #Role of Tools#
  ** Tools allow the AI to perform actions or retrieve data from external services.
  ** Example: A tool that fetches open pull requests from GitHub.
  ** Tools act as connectors between the AI model and external APIs.

* Challenge Without MCP
  ** Developers must manually create tools for every API feature.
  ** Example: GitHub has many features.
  *** Repositories
  *** Pull requests
  *** Issues
  *** Projects
  ** Each feature would require separate tool schemas and function implementations.
  ** Developers must write, test, and maintain all this code.

* How MCP Solves the Problem
  ** MCP shifts tool definition and execution to an MCP server.
  ** Developers no longer need to implement all tools themselves.
  ** The MCP server provides ready-made tools for external services.

* MCP Server Concept
  ** An MCP server acts as an interface to an external service.
  ** It wraps service functionality into tools.
  ** Example: A GitHub MCP server exposes GitHub features as tools.

* Benefits of MCP
  ** Reduces development effort.
  ** Eliminates the need to build and maintain many integrations.
  ** Provides standardized access to external services.

* Common Question: Who Builds MCP Servers
  ** Anyone can create an MCP server.
  ** Often created by service providers themselves.
  *** Example: AWS could provide an official MCP server.
  *** The server would include tools for interacting with AWS services.

* Common Question: Why Use MCP Instead of Calling APIs Directly
  ** Direct API usage requires developers to build tools themselves.
  *** Define tool schemas.
  *** Implement functions.
  ** MCP servers provide these tools already implemented.
  ** This saves development time and effort.

* Common Misconception: MCP vs Tool Use
  ** Some people think MCP and tool use are the same.
  ** In reality, they are complementary concepts.
  *** Tool use: AI calling functions to perform actions.
  *** MCP: A system that provides pre-built tools.

* Key Insight
  ** MCP is about delegating the work of creating and maintaining tools.
  ** Instead of developers building tools, MCP servers provide them.

* MCP Client
** The MCP client enables communication between your server and an MCP server.
** It acts as the access point for all tools provided by the MCP server.
** Your server interacts with tools through the MCP client.

* Transport Agnostic Communication
** MCP does not depend on a specific communication protocol.
** Client and server can communicate using different transports.
*** Standard Input / Output (stdio)
**** Common when client and server run on the same machine.
*** HTTP
*** WebSockets
*** Other communication protocols

* Message-Based Communication
** MCP client and server communicate by exchanging structured messages.
** All message types are defined in the MCP specification.

* Tool Discovery Messages
** ListToolsRequest
*** Sent from MCP client to MCP server.
*** Requests a list of available tools.
** ListToolsResult
*** Sent from MCP server to MCP client.
*** Contains the list of tools provided by the server.

* Tool Execution Messages
** CallToolRequest
*** Sent from MCP client to MCP server.
*** Requests execution of a specific tool with arguments.
** CallToolResult
*** Sent from MCP server to MCP client.
*** Contains the output produced by the tool.

* End-to-End Example Flow (#above#)

** Step 1: User Request
*** A user sends a query to the application server.
*** Example: “What repositories do I have?”

** Step 2: Server Needs Tool Information
*** The server must provide available tools when querying the AI model.
*** The server asks the MCP client to retrieve the list of tools.

** Step 3: Tool List Retrieval
*** MCP client sends ListToolsRequest to MCP server.
*** MCP server responds with ListToolsResult containing available tools.
*** MCP client returns the tool list to the server.

** Step 4: Request Sent to AI Model
*** The server sends a request to the AI model (Claude).
*** The request includes:
**** The user’s query.
**** The list of available tools.

** Step 5: AI Decides to Use a Tool
*** The AI analyzes the request and available tools.
*** It determines that a tool must be executed.
*** The AI responds with a tool use instruction.

** Step 6: Server Requests Tool Execution
*** The server receives the tool request from the AI.
*** The server asks the MCP client to run the tool.

** Step 7: Tool Execution Through MCP
*** MCP client sends CallToolRequest to the MCP server.
*** MCP server executes the requested tool.

** Step 8: External Service Call
*** The MCP server calls an external service.
*** Example: GitHub API to retrieve repositories.

** Step 9: Tool Result Returned
*** GitHub returns the requested data.
*** MCP server wraps the data in CallToolResult.
*** MCP server sends the result to the MCP client.
*** MCP client forwards the result to the server.

** Step 10: Result Sent Back to AI
*** The server sends the tool result to the AI model.
*** The result is included as part of a message.

** Step 11: Final AI Response
*** The AI uses the tool result to generate the final answer.
*** Example: “Your repositories are…”

** Step 12: Response Delivered to User
*** The server sends the final response back to the user.

* Key Insight
** MCP separates responsibilities across multiple components.
** The MCP client handles communication.
** The MCP server handles tool execution.
** External services provide the actual data.

=== Defining tools with MCP

* MCP server setup for CLI chatbot
  ** Extend existing CLI chatbot with MCP server functionality
  ** Add tools to enable document operations

* MCP server implementation
** Create server in `mcpserver.py` file
** Use *MCP Python SDK* for quick setup. Single line creates MCP server instance

* In-memory document storage
** Store documents in a dictionary (`docs`)
** Keys represent document IDs
** Values contain document content

* Tool creation using MCP SDK
** Use `@mcp.tool()` decorator to define tools
** Provide tool name and description
** Define function with typed arguments
*** Use `pydantic.Field` for argument metadata
** SDK auto-generates JSON schema
*** No need to manually write schemas

* Tool 1: Read document
** Purpose: return document content
** Input: document ID
** Logic:
*** Check if document exists in `docs`
*** Raise error if not found
*** Return document content if found

* Tool 2: Edit document
** Purpose: modify document content
** Inputs: document ID, old string, new string
** Logic:
*** Validate document exists
*** Perform string replace operation
*** Update document in dictionary

* Error handling
** Use validation to check document existence
** Raise `ValueError` for invalid document ID

* Key benefit of MCP SDK
** Simplifies tool definition process
** Automatically handles schema generation
** Reduces complexity compared to manual JSON schema writing

* Final outcome
** MCP server created with two tools
*** Read document tool
*** Edit document tool
  ** Ready for integration with Claude or other AI systems

* *MCP Resources Concept*
** Resources expose data from MCP server to client
** Used for read operations
** Each resource represents one type of data access

* *Feature Goal*
** Allow users to mention documents using `@document_name`
** Automatically fetch document content
** Inject content into prompt sent to Claude

* *Two Key Features*
** Autocomplete on `@` typing
*** Show list of available documents
** Fetch document content on submit
*** Insert content into prompt before sending to Claude

* *#Why This Approach#*
** Avoid relying on Claude to fetch document data
** Provide context upfront in the prompt
** Improves response accuracy and control

* *Types of Resources*
** Direct (Static) Resources
*** Fixed URI
*** Example: `docs://documents`
** Templated Resources
*** Dynamic URI with parameters
*** Example: `docs://documents/\{doc_id\}`
*** Parameters passed as function arguments

* *Resource Flow*
** User types `@`
*** Client requests list of documents
*** MCP client sends read request with URI
*** MCP server returns document list
** User submits message with mention
*** Client requests specific document
*** MCP server returns document content

* *Resource Implementation*
** Resource 1: List Documents
*** URI: `docs://documents`
*** MIME type: `application/json`
*** Returns list of document IDs (names)
*** Used for autocomplete

** Resource 2: Fetch Document
*** URI: `docs://documents/\{doc_id\}`
*** MIME type: `text/plain`
*** Returns document content
*** Validates document existence

* *MIME Type Usage*
** Indicates format of returned data
** `application/json` → structured data
** `text/plain` → raw text
** Helps client decide how to process response

* *SDK Behavior*
** Automatically serializes return values to string
** JSON responses must be deserialized by client
** Template parameters mapped to function arguments

* *Error Handling*
** Check if document exists
** Raise error if document not found

* *Testing with MCP Inspector*
** Start server using `uv run mcp dev mcp_server.py`
** Connect via browser (default port 6274)
** View available resources
*** Direct resources
*** Resource templates
** Test document list
*** Returns JSON string
** Test document fetch
*** Returns plain text content

* *Key Takeaways*
** Use resources for structured data access
** Separate resources for list and detail operations
** Use templated resources for dynamic queries
** Always define appropriate MIME type

=== Defining prompts

* Prompts in MCP server
  ** Used to define pre-built, high-quality instructions for specific tasks
  ** Help improve results compared to user-written prompts
  ** Designed for reuse across client applications

* Feature implemented: Slash command (`/format`)
  ** Typing `/` shows available commands
  *** Currently supports only `format`
  ** Selecting `format` asks for a document ID
  *** Example: `report.pdf`
  ** Executes a prompt to reformat document into Markdown

* Goal of the feature
  ** Convert plain text documents into Markdown format
  ** Use Claude to rewrite content
  ** Improve output quality using a well-designed prompt

* Why prompts are needed
  ** Users can manually ask for formatting
  *** Example: “convert this to markdown”
  ** Results may vary in quality
  ** Predefined prompts ensure:
  *** Better consistency
  *** Higher quality output
  *** Tested and optimized behavior

* Role of MCP server
  ** Stores specialized prompts for its domain
  *** Example: document management
  ** Exposes prompts to client applications
  ** Avoids duplicating prompt logic in clients

* Prompt definition in MCP
  ** Uses a prompt decorator
  *** Includes name and optional description
  ** Accepts input parameters
  *** Example: `docId` (string)
  ** Returns a list of messages
  *** User and assistant messages

* Prompt implementation flow
  ** Receive document ID
  ** Ask Claude to:
  *** Read document using a tool
  *** Convert content to Markdown
  *** Optionally save updated content

* Code structure highlights
  ** Import base prompt message class
  ** Define function with type annotations
  ** Write a detailed prompt string
  ** Return messages using base user message

* Testing the prompt
  ** Run MCP inspector
  ** Connect to server
  ** Navigate to prompts section
  ** Select `format` prompt
  ** Enter document ID
  ** Retrieve generated messages
  ** Send messages to Claude for execution

* Key benefit
  ** Prompts are:
  *** Pre-tested
  *** Evaluated
  *** Tailored for specific use cases
  ** Provide better user experience than ad-hoc prompts
